## Baseline discovery & defensive commands (what every pro must master)
These commands are for inventory, triage, and detection tuning in authorized labs/staging.

Network & host reachability
- ping (basic reachability)
  - ping -c 4 10.0.0.10

Port/service discovery (non-destructive; tune timing)
- Full TCP SYN + service/version scan (lab only):
  - nmap -sS -sV -p- -T3 --open -oA nmap_full 10.0.0.10
- Focused web/service scan:
  - nmap -sC -sV -p 80,443,8080 -T3 -oN nmap_web 10.0.0.10

HTTP / TLS inspection
- View response headers:
  - curl -I -sS https://10.0.0.10
- Fetch full response and headers:
  - curl -sS -D headers.txt -o body.html https://10.0.0.10/path
- Inspect TLS certificate:
  - echo | openssl s_client -connect 10.0.0.10:443 2>/dev/null | openssl x509 -noout -text

Directory / endpoint discovery (low-rate)
- ffuf (directory discovery; throttle requests)
  - ffuf -u http://10.0.0.10/FUZZ -w wordlist.txt -p 0.2 -t 10
  - (Use conservative concurrency/rate flags; treat results as candidates to verify manually)

Packet capture & analysis
- Capture traffic to PCAP:
  - sudo tcpdump -i eth0 host 10.0.0.10 -w lab_capture.pcap
- Extract HTTP requests with tshark:
  - tshark -r lab_capture.pcap -Y 'http.request' -T fields -e frame.time -e ip.src -e http.host -e http.request.uri

Proxy-based inspection (Burp/ZAP)
- Configure browser proxy to 127.0.0.1:8080 and install Burp CA cert for HTTPS inspection.
- Use Repeater to replay safe, synthetic requests and to inspect flows. Do not use Burp to run destructive payloads in production.

Log inspection
- Tail web server logs:
  - sudo tail -F /var/log/nginx/access.log /var/log/nginx/error.log
- Filter logs by IP:
  - awk '$1 == "10.0.0.5"' /var/log/nginx/access.log | tail

Code & dependency scanning
- Semgrep (SAST useful in class):
  - pip install semgrep
  - semgrep --config=p/ci ./src
- Trivy (container scanning):
  - trivy image --severity HIGH,CRITICAL your-image:latest

## Safe offensive / emulation commands & practices
Emphasize attacker mindset while using only benign, demonstrative actions and synthetic markers.

Safe marker POST (prove reachability & logging without exfiltration)
- curl -sS -X POST 'http://10.0.0.10/api/echo' -H 'Content-Type: application/json' -d '{"marker":"CLASS-<student-id>-2026"}'
- Rationale: demonstrates that input reaches server and is logged; use synthetic only.

Token lifecycle checks (staging)
- Use test account to request token (API-specific; example curl):
  - curl -sS -X POST 'https://staging.example/api/auth' -H 'Content-Type:application/json' -d '{"user":"test","pass":"test"}'
- Test expired token handling:
  - curl -i -H "Authorization: Bearer <expired-token>" https://staging.example/api/user

API schema inspection (OpenAPI)
- Download OpenAPI spec and inspect:
  - curl -sS https://staging.example/openapi.json | jq '.paths'

Contract fuzzing (staging only)
- Use contract-based fuzzer or limited mutation tool at low rate; e.g., use a fuzzer that respects OpenAPI, or a local script that mutates JSON fields and sleeps between requests.

Important: Do NOT publish any exploit payloads. When demonstrating high-risk concepts, use instructor-run, isolated demos or videos and ensure no exploit strings are given to students.

## Detection & monitoring — IDS, SIEM, rules (commands & examples)
Start Suricata (lab monitoring VM)
- sudo suricata -c /etc/suricata/suricata.yaml -i eth0
Tail fast log
- tail -f /var/log/suricata/fast.log

Example Suricata rule (lab use only — detection of high SYN rate)
```
alert tcp any any -> $HOME_NET any (msg:"LAB: possible SYN scan"; flags:S; detection_filter: track by_src, count 20, seconds 60; sid:1000001; rev:1;)
```
Explanation: alerts if a source sends 20 SYNs to hosts in HOME_NET within 60 seconds. Tune counts/intervals for lab size.

SIEM / Kibana KQL examples (assumes logs ingested)
- Excessive 401s from single IP (last 10 min):
  - event.action : "authentication_failure" and source.ip : "10.0.0.5" and @timestamp >= now-10m
- Admin endpoint accessed (last 1 hour):
  - event.dataset:"nginx.access" and url.path : "/admin" and @timestamp >= now-1h

Alerting idea: correlate application logs (auth failures), WAF logs, and network IDS alerts to reduce false positives.

## Lab exercises (authorized, non-destructive) — step-by-step (commands included)
Each exercise uses isolated lab network and synthetic data.

Exercise 1 — Reconnaissance & IDS tuning
- Objective: observe how scanning appears to IDS and tune detection rules.
- Steps:
  1. Start monitoring VM with Suricata:
     - sudo suricata -c /etc/suricata/suricata.yaml -i eth0
  2. From attacker VM, run controlled nmap scan:
     - nmap -sS -sV -p 22,80,443 -T3 --open -oN nmap_lab 10.0.0.10
  3. On monitoring VM, correlate:
     - tail -n 200 /var/log/suricata/fast.log
  4. Tune rule thresholds and re-run scan to see differences.
- Deliverables: nmap output, Suricata alert excerpts, short write-up on tuning.

Exercise 2 — HTTP header hardening
- Objective: identify missing security headers and implement fixes.
- Steps:
  1. Inspect headers:
     - curl -I -sS https://10.0.0.10
  2. In staging Nginx, add headers (example):
     - add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
     - add_header X-Frame-Options "DENY" always;
     - add_header Content-Security-Policy "default-src 'self';" always;
  3. Reload Nginx and verify:
     - sudo nginx -s reload
     - curl -I -sS https://10.0.0.10
- Deliverables: before/after header outputs, explanation of each header.

Exercise 3 — Auth lifecycle & logging
- Objective: validate token revocation, expiry handling, and logging.
- Steps:
  1. Obtain test token (adapt to app):
     - curl -sS -X POST 'https://staging.example/api/auth' -H 'Content-Type: application/json' -d '{"user":"test","pass":"test"}'
  2. Use expired token to access resource:
     - curl -i -H "Authorization: Bearer <expired-token>" https://staging.example/api/user
  3. Inspect application logs for rejection entry:
     - sudo tail -n 200 /var/log/app/app.log | grep "<token-id or user>"
- Deliverables: sanitized request/response, log snippet, remediation suggestions.

Exercise 4 — IDOR detection with synthetic data
- Objective: demonstrate ownership checks are enforced.
- Steps:
  1. Create resource as user A (staging):
     - curl -sS -X POST 'https://staging.example/api/resources' -H 'Authorization: Bearer <token-A>' -d '{"name":"class-demo"}'
  2. Attempt access with user B to same resource ID:
     - curl -i -H "Authorization: Bearer <token-B>" https://staging.example/api/resources/<id>
  3. Confirm response is 403/404 and log entry exists.
- Deliverables: request/response, server logs, updated server-side ownership check PR (instructor-reviewed).

Exercise 5 — Contract fuzzing (staging only)
- Objective: find input handling issues without exploits.
- Steps:
  1. Obtain OpenAPI spec of staging API:
     - curl -sS https://staging.example/openapi.json -o openapi.json
  2. Use contract-based fuzzer (configured to low rate) to send mutated JSON bodies. (Tool choice taught in class; run with <=1 req/s)
  3. Monitor application logs for 5xx errors or stack traces and report sanitized evidence.
- Deliverables: fuzzer run summary, log excerpts, suggested fixes (input validation/canonicalization).
